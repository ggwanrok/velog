<li><p>클라우드 네이티브 어플리케이션 개발은 기존 개발 공정과 다르게 마이크로서비스 도출.설계.개발, 마이크로 서비스 아키텍처 설계.구축, 빌드.배포 등의 태스크가 포함된다.</p>
</li>
<li><p>현 시점에선 해당 방법론의 절차만 간단하게 알아보고 12 factors를 알아볼 것이다.</p>

<li><p>각 단계에 대한 msa 도출 과정에 대해서는 다음 시간에 자세하게 다룰 것이다.</p>

<li>DDD 설계를 통해서 사용 역할군에 대한 논리적 분리를 진행하는 것이 중요하다.</li>
</ul>

<h2 id="msa의-개발-원칙">MSA의 개발 원칙</h2>
<h3 id="4가지-대분류">4가지 대분류</h3>
<p><img alt="" src="https://velog.velcdn.com/images/klmcw1004/post/3fac0f7a-1fe0-4281-934b-3d5818ed3f6d/image.png" /></p>
<p>잘 만들어진 MSA는 위의 4가지 대분류를 만족시킨다.</p>
<p>이걸 어떻게 알고 파악해서 적용하는 것이냐? 에 대한 의문에 대한 답은 12 factors이다.
대분류를 12가지로 나누어 각 속성에 만족하도록 msa를 구상한다.</p>
<h3 id="12-factors">12 Factors</h3>
<h4 id="코드-베이스">코드 베이스</h4>
<p><img alt="" src="https://velog.velcdn.com/images/klmcw1004/post/62786efa-374f-4157-a526-a479d4547b85/image.png" /></p>
<ul>
<li>버전 관리가 가능한 하나의 코드베이스와 이를 이용한 여러 배포를 통해서 어플리케이션을 관리한다.</li>
</ul>
<h4 id="종속성">종속성</h4>
<p><img alt="" src="https://velog.velcdn.com/images/klmcw1004/post/bebe0d67-7c2b-4b2d-af15-5cb446132342/image.png" /></p>
<h4 id="설정">설정</h4>
<p><img alt="" src="https://velog.velcdn.com/images/klmcw1004/post/806cf791-6722-4a25-bf43-5e88e67f6d17/image.png" /></p>
<ul>
<li>여러 배포에 따른 여러 환경파일을 구성해야 한다.</li>
</ul>
<h4 id="백엔드-서비스">백엔드 서비스</h4>
<p><img alt="" src="https://velog.velcdn.com/images/klmcw1004/post/c1bd6140-fe9b-400e-bd21-ef4e5f62b3ab/image.png" /></p>
<ul>
<li>결국 컨테이너를 구성하여 사용할 때, 컨테이너에서 취급하고자 하는 데이터는 백엔드에서 와야 한다.<ul>
<li>즉, 백엔드 서비스 자체를 리소스 취급을 해야한다는 것이고 여기 종속되면 안되고 전환이 가능해야 한다.</li>
</ul>
</li>
</ul>
<h4 id="빌드-릴리즈-실행">빌드, 릴리즈, 실행</h4>
<p><img alt="" src="https://velog.velcdn.com/images/klmcw1004/post/304608cc-bd6b-4914-b8f3-a73804cfe606/image.png" /></p>
<ul>
<li>프로그램의 배포는 3단계에 걸쳐서 진행된다.<ol>
<li>코드베이스를 빌드한 뒤</li>
<li>설정파일과 함께 릴리즈하여</li>
<li>릴리즈된 프로세스들의 집합을 시작하여 실행한다.</li>
</ol>
</li>
</ul>
<h4 id="프로세스">프로세스</h4>
<p><img alt="" src="https://velog.velcdn.com/images/klmcw1004/post/7638ce38-e71a-4b85-9cb5-80f92d50b427/image.png" /></p>
<ul>
<li>기본적으로 프로세스는 stateless 상태여야 한다.<ul>
<li>다른 컨테이너에 연결하든 뭘 하든 지장이 없어야 하기 때문이다.</li>
</ul>
</li>
</ul>
<h4 id="포트-바인딩">포트 바인딩</h4>
<p><img alt="" src="https://velog.velcdn.com/images/klmcw1004/post/aef22a60-8c55-47c0-81dc-cfa04b9265d0/image.png" /></p>
<ul>
<li>독립적인 실행 단계에서 각 컨테이너들을 포트로 판단해서 사용할 수 있게 한다.</li>
</ul>
<h4 id="동시성">동시성</h4>
<p><img alt="" src="https://velog.velcdn.com/images/klmcw1004/post/7246cee2-ff79-4ae5-845c-c59a8dc2d27b/image.png" /></p>
<h4 id="폐기-가능">폐기 가능</h4>
<p><img alt="" src="https://velog.velcdn.com/images/klmcw1004/post/c7f2c0ed-bda7-46f2-9866-2cce8b40d80b/image.png" /></p>
<ul>
<li>프로세스는 갑작스러운 종료 상태에도, 동작에 손실이 발생하지 않도록 <strong>graceful shutdown</strong>을 지원해야 한다. (안정성 극대화)</li>
<li>또한, 종료되는 프로세스로 인해, 다른 프로세스에 영향이 끼치면 안된다.</li>
</ul>
<h4 id="devprod-일치">dev/prod 일치</h4>
<p><img alt="" src="https://velog.velcdn.com/images/klmcw1004/post/89b2fd0f-eb86-43a6-ad25-e4af3e49b87d/image.png" /></p>
<h4 id="로그">로그</h4>
<p><img alt="" src="https://velog.velcdn.com/images/klmcw1004/post/3e1de28e-8704-47c3-a30c-0961f9021e92/image.png" /></p>
<ul>
<li>msa 구조 특성상 통합 테스트에 어려움이 발생한다.<ul>
<li>이를 보완하기 위해선 로그처리를 통해서 각 컨테이너의 상태를 잘 파악할 수 있어야 한다.</li>
</ul>
</li>
</ul>
<h4 id="admin-프로세스">admin 프로세스</h4>
<p><img alt="" src="https://velog.velcdn.com/images/klmcw1004/post/c93d1e64-ef30-403d-995b-c8dc552bfa6a/image.png" /></p>
<ul>
<li>권한이 잘 나뉘어져 있어야 한다.<ul>
<li>마이크레이션이나 유지보수와 같은 작업들을 권한 검증이 된 상황에서만 진행되어야 한다.</li>
</ul>
</li>
</ul>
<h3 id="정리">정리</h3>
<p>위와 같은 12 규칙을 통해서 4가지 대원칙을 준수할 수 있도록 msa 설계작업을 진행할 수 있다.</p>
