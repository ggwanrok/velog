<h1 id="정리">정리</h1>
<p>이번 CI/CD 구축을 통해 솔브싱크 프로젝트는 다음과 같은 배포 흐름을 갖게 되었다.</p>
<pre><code class="language-text">dev 브랜치 push
  ↓
GitHub Actions 실행
  ↓
Gradle test / bootJar
  ↓
Docker image build
  ↓
GHCR push
  ↓
EC2 SSH 접속
  ↓
docker compose pull app
  ↓
docker compose up -d
  ↓
최신 Spring Boot 애플리케이션 실행</code></pre>
<p>이번 작업에서 핵심은 단순히 Dockerfile을 작성한 것이 아니라, GitHub Repository, GitHub Actions, GHCR, EC2, Docker Compose를 연결해 하나의 배포 파이프라인을 구성했다는 점이다.</p>
<p>각 구성 요소의 역할은 다음과 같다.</p>
<pre><code class="language-text">Dockerfile
→ Spring Boot jar를 Docker 이미지로 만드는 파일

deploy.yml
→ 빌드, 테스트, 이미지 push, EC2 배포를 자동화하는 GitHub Actions workflow

GHCR
→ Docker 이미지를 저장하는 Registry

EC2
→ Docker Compose 기반으로 컨테이너를 실행하는 서버

docker-compose.yml
→ app, mysql, rabbitmq, dozzle 컨테이너 실행 정의

application.yml
→ 운영 환경 설정 파일

GitHub Secrets
→ SSH key, GHCR PAT 등 민감 정보 관리</code></pre>
<p>이 구조를 통해 EC2에서 직접 빌드하지 않고, GitHub Actions에서 검증된 이미지만 배포할 수 있게 되었다.</p>
<p>또한 MySQL과 RabbitMQ를 Docker Compose로 함께 관리하면서, 애플리케이션 실행 환경의 재현성과 배포 자동화를 높일 수 있었다.</p>
<p>아직 무중단 배포, DB 마이그레이션, healthcheck, SSH 보안 강화 등 개선할 부분은 남아 있지만, 소규모 프로젝트의 초기 운영 환경으로는 충분히 실용적인 CI/CD 구조를 구축할 수 있었다.</p>
