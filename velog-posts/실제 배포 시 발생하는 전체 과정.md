<h1 id="실제-배포-시-발생하는-전체-과정">실제 배포 시 발생하는 전체 과정</h1>
<p>개발자가 <code>dev</code> 브랜치에 push하면 다음 일이 순서대로 발생한다.</p>
<pre><code class="language-text">1. GitHub의 dev 브랜치가 갱신된다.

2. GitHub Actions가 push 이벤트를 감지한다.

3. Deploy workflow가 실행된다.

4. GitHub Actions Runner가 생성된다.

5. Repository 코드가 checkout된다.

6. Java 17 환경이 설정된다.

7. Gradle clean, test, bootJar가 실행된다.

8. 테스트가 실패하면 배포가 중단된다.

9. 테스트와 빌드가 성공하면 build/libs/*.jar 파일이 생성된다.

10. Dockerfile을 기반으로 Docker 이미지가 생성된다.

11. 이미지에 latest 태그와 github.sha 태그가 붙는다.

12. GitHub Actions가 GHCR에 로그인한다.

13. Docker 이미지가 GHCR에 push된다.

14. GitHub Actions가 EC2에 SSH로 접속한다.

15. EC2에서 GHCR에 로그인한다.

16. EC2에서 docker compose pull app을 실행한다.

17. 최신 app 이미지를 GHCR에서 가져온다.

18. docker compose up -d를 실행한다.

19. 기존 app 컨테이너가 새 이미지 기반으로 교체된다.

20. MySQL과 RabbitMQ는 volume을 유지한 상태로 계속 동작한다.

21. Spring Boot 애플리케이션이 최신 버전으로 실행된다.</code></pre>
<p>결과적으로 개발자는 <code>dev</code> 브랜치에 코드를 반영하는 것만으로 서버 배포까지 자동화할 수 있다.</p>
